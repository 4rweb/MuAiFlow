# Cross-Review Prompt

Use this prompt when one AI needs to review a plan generated by another AI.

## Instructions

You are reviewing a plan generated by another AI. Your role is to **improve the plan** with concrete evidence — not to emit generic opinions.

Read the exact plan path provided by the human. Plans usually live at `.ai/plans/tracked/YYYY-MM-DD-title.md` or `.ai/plans/local/YYYY-MM-DD-title.md`; older plans may still live directly under `.ai/plans/`.

## Required Review Criteria

### 1. Verify real files
- **EVERY file path cited in the plan must be verified.** Use read tools to confirm the file exists and that the proposed change makes sense in the context of the current code.
- If the plan says "modify `src/modules/users/users.service.ts`", **read the file** and confirm the change is compatible with existing code.
- If the file doesn't exist, mark it as a **blocker**.

### 2. Cite line numbers when pointing out risks
- Never just say "might break". Say: "when modifying `users.service.ts:47`, the query `db.select()` doesn't include the `ownerId` filter, which may expose data from other users."
- Always include: `file:line` + relevant snippet + why it's a risk.

### 3. Separate blockers from suggestions
- **BLOCKER**: something that prevents safe execution — must be resolved before recommending for human approval.
- **SUGGESTION**: optional improvement — the plan works without it, but would be better with it.

### 4. Verify required tests
- For each task, ask: "what test validates this works?"
- If the plan doesn't mention a test for a business logic task, mark it as a blocker.
- Check if existing tests cover the paths being modified.

### 5. Verify security risks
- Does the plan change authentication or authorization? Verify it follows the project's security conventions in `CLAUDE.md` / `AGENTS.md`.
- Does the plan add `.env` or secrets? Verify no secret goes into code or commits.
- Does the plan make DB queries? Verify SQL injection and OWASP top 10.
- Does the plan expose new endpoints? Verify they have proper authentication.

### 6. Verify migration and data
- Does the plan create/modify a table? Confirm the migration is planned.
- Is there a data loss risk? If so, mark as **blocker** and suggest a safe strategy.
- Are new env vars documented in `.env.example`?

## Validation Checklist

Use this checklist. Each item must have **evidence** (file read, line cited) or justification for why it doesn't apply.

- [ ] **Completeness** — Does the plan cover all aspects of the problem? Cite what's missing.
- [ ] **Files verified** — EVERY cited path was read and confirmed. List each file verified.
- [ ] **Order and dependencies** — Are tasks in the correct sequence? Are explicit dependencies correct?
- [ ] **Project standards** — Follows `CLAUDE.md` / `AGENTS.md`? Cite the rule and line if violated.
- [ ] **Tests** — Does each task have a test strategy? Existing tests won't break?
- [ ] **Security** — No secret exposure, protected endpoints, safe queries.
- [ ] **Migration** — Safe schema changes, no data loss, rollback viable.
- [ ] **Scope** — Nothing ambiguous. Done criteria are objective and measurable.
- [ ] **Rollback** — Rollback strategy is viable and complete?

## Feedback Format

```markdown
## Plan Validation: [title]

**Reviewer**: [AI name]
**Date**: YYYY-MM-DDTHH:mm
**Files verified**: [list of files read during review]

---

### Blockers (must be resolved before human approval)
- [ ] [BLOCKER] [description] — `file:line` — [how to resolve]

### Suggestions (optional improvements)
- [ ] [SUGGESTION] [description] — [why it would be better]

### Identified Risks
| Risk | Severity | File:line | Detail |
|------|---------|-----------|--------|
| [risk] | high/medium/low | `path:NN` | [explanation with evidence] |

### Verdict
- [ ] **Recommended for human approval** — no pending blockers
- [ ] Needs adjustments — blockers listed above must be resolved
```

## Principles

- **Evidence over opinion** — all feedback must reference a file/line or project rule.
- Be constructive — suggest solutions, don't just point out problems.
- Respect the style of the AI that created it — suggest, don't rewrite the entire plan.
- If the plan is good, say so with evidence — "I verified all 5 cited files, they all exist and the proposed changes are compatible."

## ⚠️ Status Blocking Rule

- You CAN change status to: `AI_REVIEWED` or `CHANGES_REQUESTED`
- You **NEVER** can change status to `HUMAN_APPROVED`
- You **NEVER** can fill `human_approved_by` or `human_approved_at` in the frontmatter
- You **NEVER** can fill the "Approval (human)" section
- Your verdict is always **"recommended for human approval"** — never "approved"
